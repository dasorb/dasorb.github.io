SpringIoC控制反转

IoC概念

控制反转：是一种通过描述并通过第三方去产生或获取特定对象的方式，由主动去创建对象，改为需要时去被动获取对象，不会关注对象是如何产生，只要在需要的时刻能够获取到这个对象。

IoC容器主要基于两个接口：BeanFactory和ApplicationContext

BeanFactory接口设计的方法：

  • getBean()，使用多种方式去获取容器中的Bean
  • isSingleton()，判断是否是单例，isPrototype()判断相反
  • getAliases()，获取bean的别名

容器初始化步骤

Bean的初始化分为3步：

  1. Resource根据配置进行定位
  2. BeanDefinition的载入，根据配置获取对应POJO，以生成实例的过程
  3. BeanDefinition的注册，将上步载入的POJO，往IoC容器中注册。

执行上述3步之后，未完成对象的依赖注入，通过lazy-init属性来控制对象是否自动初始化，，默认false，true，只有获取这个Bean时，才执行初始化。

Bean的生命周期

程序启动时（如果实现了接口）：

  • BeanNameAware接口的setBeanName()

  • BeanFactoryAware接口的setBeanFactory()

  • ApplicationContextAware接口的setApplicationContext()，并且spring容器是一个ApplicationContext接口的实现类。

  • BeanPostProcessor的postProcessBeforeInitialization() bean开始初始化执行，针对所有bean

  • BeanFactoryPostProcessor接口的afterPropertiesSet ()

  • Bean自定义初始化方法，init自定义初始化方法

  • BeanPostProcessor接口的postProcessAfterInitialization() bean初始化并依赖注入完成之后，针对所有bean

容器关闭时：

  • DisposableBean接口的detory() 针对IoC容器的销毁方法

  • Bean自定义销毁方法 destroy()自定义销毁方法

注意：

  1. 当一个Bean实现了上述的接口，我们只要在容器中定义它们就可以了。
  2. 一个bean的自定义初始化、销毁方法，使用init-method和destroy-method属性去指定。

Bean的依赖注入

依赖注入可分3种方式：

  1. 构造器注入
    • 在bean标签中使用<constructor-arg>标签，index代表参数顺序，value代表值。
  2. setter注入
  3. 接口注入

装配Bean的方式：

  1. 在xml中显示配置
  2. 在java的接口和类中实现配置
  3. 隐式bean的发现机制和自动装配原则

通过XML进行配置

装配简单值：

  • id属性代表这个bean在容器中的唯一编号
  • class指定类全限定名
  • <property>便签定义类的属性

装配集合：

  • list，在property中使用<list><value></value></list>标签。使用<ref/>来引入对象
  • map，在property中使用<map><entry key="" value=""/></map>标签，使用value-ref=""属性来引入对象
  • properties,在property中使用<props><prop key="">value</prop></props>标签
  • set，在property中使用<set><value></value></set>标签
  • array，在property中使用<array><value></value></array>标签

通过注解来装配

通过两种方式来让springIoC来发现Bean：组件扫描，自动装配

@Component装配Bean

  • 括号内默认写id名称，不写则默认类名首字母小写为id。
  • @value,代表值的注入

组件扫描：

  • 使用@ComponentScan修饰一个类，默认扫描这个类所在包和子包下的所有注解标识的Bean
  • 配合 AnnotationApplicationContext实现类，进行扫描。
  • 配置项basePackages，可以加载多个java包，即数组。配置项basePackageClasses，可以配置多个类的全限定名数组。
  • 也可以通过<context:component-scan base-package='com...'/>xml配置进行组件扫描

自动装配：

  • @Autowired，配置项required，默认为ture；为false代表自动注入失败时，不抛出异常，该属性不必要，有空指针危险。
  • 对于相同的接口，有多个实现类，加载时为了消除歧义性，有两种方式解决：
    • 在实现类上使用@Primary告诉容器，优先自动装配该类。
    • 配合@Autowired，使用@Qualifier()告诉容器按名称自动装配。

@Bean作用于方法：

  • 对于第三方类，将@Bean注解作用于方法，将方法的返回对象自动装配到容器中。
  • 配置项：name，配置BeanName；autowire，是否引用一个Bean对象；initMethod，自定义初始化方法；detoryMethod，自定义销毁方法。

配置文件的引用

一个配置类的中的配置过于复杂，spring提供@import加载其他配置类。

同样的在xml中可以通过一个配置文件引用其他配置文件：<import resource="xxx.xml">

@Profile切换环境

通过注解@Profile("")可以标注一个Bean在什么环境下才可以使用。

也可以通过xml中beans标签的profile属性来配置。

如何启动：

  1. 如果是测试代码中，使用@ActiveProfiles("")指定加载哪个Profile。
  2. 在服务器上或JVM本地启动，制定有两个参数：
    • spring.profile.active，用它去指定要加载什么环境，默认的将失效。
    • spring.profile.default，系统如果没有配置，默认启动。
  3. JVM中JAVA_OPTS="_Dspring.profiles.active=test"
  4. Web环境中：<context-param>标签或DispactchServlet的<init-param>标签

@PropertySource加载属性文件

注解方式：

  • 在类上使用注解@PropertySource(value="classpath:")加载属性文件。
  • 容器实现PropertySourcePlaceHolderConfigurer的bean可以使用占位符${}来加载配置值。
  • 也可使用代码context.getEnvironment().getProperty("xx")来加载属性值。

XML方式：

  • 使用<context-property-placeholder locatino="classpath:"/>标签加载，然后使用占位符
  • 如果一次加载多个属性文件，容器实现PropertyPlaceHodlerConfigruer的Bean，通过配置location,为其注入一个数组即可。

@Conditional根据条件装配Bean

使用注解@Conditional()去加载一个或多个实现Condition接口的条件类来判断是否装载注解修饰的类。

condition接口方法中有两个参数，一个包含spring上下文对象，一个包含该Bean的注解信息

Bean的作用域

Bean有4种作用域：

  1. singleton，单例，在整个应用只生成一个Bean
  2. prototype，原型，每次获取Bean时spring为它生成一个实例
  3. session，在Web应用中，在会话过程中只创建一个实例
  4. request，在Web应用中，一次请求spring为它创建一个实例

配置方式：

  • 使用注解@Scope()表明作用域
  • xml中在bean标签使用scope标签

使用SpringEL表达式

SpringEL的使用方式是使用#{}来使用。

使用SpringEL来获取类中的静态常量和方法，使用@Value("#{T(全类名).方法}")

面向切面编程

概念

AOP是面向切面编程，常用于数据库事务，它可以拦截一些方法，然后把各个对象组织成一个整体，在拦截的前后可以执行一些方法。对于数据库事务，可以交给AOP去进行对数据库资源的获取和关闭，事务的提交和回滚操作，从而让我们更关注于SQL代码。

注：spring在生成代理对象的过程中，对于没有接口的目标对象采用cglib动态代理，有接口的对象采用JDK代理。

面向切面的术语

切面：切面就是在一个怎么样的环境下工作，可以理解为一个拦截器，可以在你的方法前后切入代码。

通知：通知是切面开启后，切面的方法。

  • 前置通知：即before
  • 后置通知：即after，无论是否发生异常都执行。
  • 返回通知：即afterReturning，执行成功。
  • 异常通知：即afterThrowing，方法执行不成功，抛出异常后执行。
  • 环绕通知：即aroundThrowing，可以取代当前被拦截对象的方法。

切点：被切面拦截的方法就是一个切点。

连接点：一个判断条件，由它可以指定哪些是切点。

织入：生成代理对象的过程。Spring是以JDK和CGLIB动态代理来生成代理对象。

使用@ASpectJ来开发AOP功能

  1. 选择切点，即类和方法
  2. 创建切面，spring中只要使用@Aspect注解一个类，那么spring容器就会认为这是一个切面
    • @Before
    • @After
    • @AfterReturning
    • @AfterThrowing
    • @Around
  3. 连接点，使用excution正则表达式匹配切点
    • excution：代表执行方法的时候会触发
    • *代表任意类型返回值
    • 类全限定名
    • 拦截方法名
    • (..)任意参数
  4. @Pointcut修饰一个方法，并配置连接点，其他注解可以引用这个方法来解决重复书写。
  5. 最后，在配置类使用@EnableAspectJAutoProxy，代表启用AspctJ框架自动代理。
  6. 在xml中，可以使用<aop:aspectj-autoproxy />。同样效果。

mybatis-spring数据库编程整合

项目是mybatis团队为了整合spring，封装了获取SqlSessionFactory和SqlSession的步骤。

步骤：

  1. 配置化数据源。
  2. 配置SqlSessionFactoryBean。
  3. 可以选择的配置有SqlSessionTemplate，SqlSessionFactory，同时配置的情况下优先采用SqlSessionTemplate
  4. 配置Mapper
  5. 事务管理

配置数据源

推荐在xml中配置id=”dataSource”的第三方数据库连接池。

配置SqlSesionFactoryBean

工厂几乎可以mybatis全局配置的所有配置。

两个主要属性：

  • 属性dataSource 指定数据源
  • 属性configLocation 指定全局配置文件所在位置

配置MapperFactoryBean

Mybatis中是使用动态代理为接口生成实现类，Spring无法为其生成实现类，Mybatis-Spring项目提供了一个MapperFactoryBean作为中介，可以通过配置它来实现我们想要的Mapper。

两个主要属性：

  • mapperInteface是映射器的接口
  • 如果同时配置了sqlSessionFactory和SqlSessionTemplate，会启用SqlSessionTemplate，废弃SqlSessionFactory

配置MapperScannerConfigurer

主要简化上一步，一步步配置Mapper的工作量，以扫描的形式加载Mapper

主要配置属性：

  • basePackage，指定让spring自动扫描哪个包，多个包之间用逗号分隔。
  • annotationClass，如果这个类被这个主机标识，才进行扫描。
  • SqlSessionFactoryBeanName，指定在Spring中定义SqlSessionFactoryBean的Bean名称。
  • markerInterface，指定扫描时实现了什么接口就认为它是Mapper。

主要配置步骤：

  • 通过配置MapperScannerConfigurer指定扫描包路径，指定SqlSessionFactory，指定注解annotationClass
  • 通过注解@Repository，标注mapper
  • 也可以通过指定markerInterface，实现接口就认定他是mapper

spring数据库事务

事务管理器

配置事务管理器的目的是，在容器中注册transactionManager，注入数据源，告诉spring数据库事务交给事务管理器。

常用的事务管理器是：DataSourceTransactionManager。

在spring中可以使用声明式事务和编程式事务，声明式事务又分为XML配置和注解事务。

使用注解实现spring事务

步骤：

  1. 在配置类使用注解@EnableTransactionManagement，表名使用事务驱动管理器。

  2. 配置类实现接口TransactionManagementConfiguer的annotationDrivenTransactionManager方法，并将方法的返回对象注册给容器作为事务管理器。

  3. 之后在spring上下文中使用事务注解@Transactional。

  4. 注意：在使用声明式事务时，需要配置注解驱动，才能使用@Transactional

数据库相关知识

数据库事务ACID特性：

  • 原子性：一个事务要么全部完成，要么全部不完成，没有停到中间这一环节
  • 一致性：无论多少个事务，始终要保持系统是一致的情况。
  • 隔离性：它是指两个事务之间的隔离程度。
  • 持久性：事务完成后，事务对数据的更改便持久的保存到数据库中。

丢失更新:

  • 第一种丢失：两个事务，1类提交事务并扣款，2类回滚事务。
  • 第二种丢失：两个事务，1类提交事务并扣款，2类提交事务并扣款，因为事务的不同步，导致只扣一次款。
  • 为了解决丢失更新，定义事务之间的隔离级别。

隔离级别（Isolation）:

  • 脏读：允许一个事务去读取另一个事务中未提交的事务，如果一个事务产生回滚，就会产生脏读。
  • 读/写提交：一个事务只能读取另一个事务已经提交的数据，这种方式会产生不可重复读的问题。
  • 可重复读：针对数据库的同一条记录而言的，一条记录按照一个序列化，如果不是同一条记录，这种级别会产生幻读的问题。
  • 序列化： 一种让SQL按照顺序读/写的方式。

7种事务传播行为：

传播行为是指方法之间调用事务策略的问题。

例如：如果一个大事务中，有一个小事务回滚，可能回到整个事务进行回滚。

  1. REQUIRED，如果不存在事务就创建事务，如果存在就沿用之前的业务。默认
  2. SUPPORTS，如果不存在当前事务就不启用事务，如果存在就沿用。
  3. MANDATORY，方法必须在事务内运行。不存在事务就抛出异常。
  4. RERUIRES_NEW，无论是否存在当前事务，方法都会在新的事务中运行该方法。
  5. NOT_SUPPORTED，不支持事务，如果不存在当前是事务也不会创建事务；如果存在，则挂起它，直至该方法结束后才恢复当前事务。
  6. NEVER，不支持事务，只有在没有事务的环境中才能运行它。
  7. NESTED,嵌套事务，调用方法如果抛出异常只回滚它自己内部执行的SQL，而不会滚主方法的SQL
